# Phase 1 Workflow Guide

## 目的

Phase 1 (Single-Agent Development Support) の各フェーズで、入力・出力・判断基準を一貫させる。

## 1. 要件取り込み (Requirements Intake)

### 1.1 依頼テンプレート

以下のテンプレートを最小必須とする。空欄は要確認として質問対象にする。

```
# Request
## 目的 (Goal)
- 何を達成したいか

## 背景 (Context)
- なぜ必要か
- 既存の課題や関連仕様

## 仕様 (Requirements)
- 必須機能
- 期待する入出力

## 受入条件 (Acceptance Criteria)
- 成功条件を箇条書きで

## 制約 (Constraints)
- 技術/運用/期限/コスト/互換性

## 非対象 (Non-goals)
- 今回はやらないこと

## 参考資料 (References)
- 関連ファイルやURL
```

### 1.2 依頼文からのタスク分解 (プロンプト設計方針)

- 依頼を要約し、目的・受入条件・制約を抽出する。
- 変更対象と影響範囲を推定し、作業を工程別に分割する。
- 分割は「調査/実装/テスト/ドキュメント/運用」に揃える。
- 不明点は質問として列挙し、ブロッカー優先で並べる。

出力フォーマット (例):

```
## Summary
- goal:
- acceptance:
- constraints:

## Task Breakdown
1. Investigation:
2. Implementation:
3. Tests:
4. Documentation:
5. Release:

## Open Questions
- Q1:
- Q2:
```

### 1.3 曖昧さ検出の基準と質問生成ルール

曖昧さの基準:
- 目的が成果物に落ちていない (例: "改善" のみ)
- 受入条件が不在/定量化されていない
- 対象範囲や非対象が不明
- 仕様が複数解釈できる (用語の定義がない)
- 互換性/性能/セキュリティの前提がない

質問生成ルール:
- ブロッカーから順に聞く。
- 1質問1論点、短文で。
- 既知情報を前置きし、選択肢があれば提示する。

## 2. 計画 (Planning)

### 2.1 影響範囲の探索ルール

- 依頼文のキーワードで `rg` 検索し、直接対象ファイルを列挙。
- 参照/依存のリンクを辿る (import/呼び出し/設定/スキーマ)。
- 影響候補は「直接変更」「間接影響」「テスト対象」に分類する。
- 変更の境界を明示し、不要な横断変更を避ける。

### 2.2 テスト戦略の提示方針

- 仕様/契約の変更: 契約テストを最優先。
- ロジック変更: 単体テスト中心。
- I/Oや連携変更: 統合テストを追加。
- ユーザー向け変更: CLI等のE2E/手動ガイドを提案。

### 2.3 作業スコープと工数見積り

- スコープは S/M/L/XL の4段階で提示。
- 判断基準 (例):
  - S: 1-2ファイル、局所ロジック変更、既存テスト追加のみ
  - M: 3-6ファイル、軽度の設計調整、テスト追加
  - L: 広範囲、複数モジュール、仕様更新
  - XL: アーキテクチャ変更、データ移行、広範な回帰対応

## 3. 実装支援 (Implementation Support)

### 3.1 小さな差分でのパッチ生成ルール

- 1パッチ=1意図 (機能/修正/リファクタの混在を避ける)。
- 変更は最小範囲に限定し、無関係な整形を避ける。
- 安全な順序で小さく積み上げる (準備→本体→テスト)。

### 3.2 lint/format/type チェック方針

- リポジトリの既存設定に従う。
- 可能なら差分対象の最小コマンドを実行。
- 失敗時は原因と再実行手順を提示。

### 3.3 既存パターン準拠チェック

- 近傍実装 (同ディレクトリ/同機能) のスタイルに合わせる。
- 命名規則・例外処理・ログ出力の既存パターンに合わせる。
- 新規依存は最終手段とし、追加理由を明記。

## 4. 検証 (Verification)

### 4.1 影響範囲ベースのテスト選択

- 変更モジュールの既存テストを優先実行。
- 新規挙動には最小の単体テストを追加。
- 仕様変更は契約テストの更新/追加を優先。

### 4.2 ローカル実行手順のガイド化

- 実行コマンドと前提 (env vars, fixtures) を明示。
- 失敗時のログ/出力の確認ポイントを記載。

### 4.3 失敗時の原因分離と再試行

- テスト失敗は「環境」「コード」「テスト」へ分類。
- 環境: 依存や設定を再確認。
- コード: 変更範囲を狭めて再現。
- テスト: 期待値の更新や不安定性を確認。

## 5. レビュー支援 (Review Support)

### 5.1 変更点要約フォーマット

```
## Summary
- What:
- Why:
- Impact:
- Tests:
```

### 5.2 リスク抽出の観点

- 挙動変更 (既存ユーザーへの影響)
- 互換性 (API/CLI/設定/スキーマ)
- 副作用 (性能/コスト/権限/データ)

### 5.3 追加テスト提案の基準

- 変更範囲にテストが存在しない
- エッジケース/失敗経路が未検証
- 互換性やマイグレーションが絡む

## 6. PR準備 (PR Preparation)

### 6.1 PR本文テンプレート

```
## 背景

## 変更内容

## テスト

## 注意点 / フォローアップ
```

### 6.2 変更ログ/リリースノート方針

- ユーザー影響のある変更のみ記載。
- Added/Changed/Fixed/Deprecated を基準に分類。
- 破壊的変更は先頭で強調。

### 6.3 品質チェックリスト

- セキュリティ: 権限/機密情報の扱い
- 性能: ループ/クエリ/外部I/Oの増加
- 互換性: 既存設定/データ/CLIへの影響

## 7. リカバリ (Recovery)

### 7.1 再現手順フォーマット

```
## Environment
- OS/Version:
- Branch/Commit:
- Config:

## Steps
1.
2.

## Expected

## Actual

## Logs
```

### 7.2 最小修正案の提示ルール

- 影響範囲を最小化し、挙動回復を優先。
- 新規テストは最小1件で再発防止。
- 大規模リファクタは次フェーズに分離。

### 7.3 ロールバック方針

- 即時復旧はリバート or 設定フラグの無効化。
- データ変更がある場合は逆マイグレーション手順を提示。
- 運用向けに復旧確認チェックを添付。
