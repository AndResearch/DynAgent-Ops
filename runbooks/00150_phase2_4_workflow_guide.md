# Phase 2-4 Workflow Guide

## 目的

Phase 2-4 (Multi-Agent Coordination / Maintenance and Operations / Autonomous Operations) の実行指針を詳細化する。

## 2. Multi-Agent Coordination

### 2.1 役割設計と責務

| Role | 主な責務 | 主要アウトプット |
| --- | --- | --- |
| Planner | 依頼の分解、スコープ定義、計画提示 | 作業計画、影響範囲、テスト戦略 |
| Implementer | 変更実装と最小テスト | パッチ、最小テスト結果 |
| Reviewer | 変更内容のレビュー支援 | リスク評価、追加テスト提案 |
| QA | 検証計画と実行 | テスト結果、再現手順 |
| Release | PR/リリース準備 | PR本文、リリースノート案 |

### 2.2 共有状態 (Shared State)

- タスクボード: 進行状況 (Todo/Doing/Done) と担当を明示。
- アクティビティログ: 変更履歴と判断理由を記録。
- コスト/予算: 予算上限、消費額、警戒ラインを可視化。

### 2.3 クロスレビューと合意形成

- Implementerの変更はReviewer/QAが独立に評価。
- 相違がある場合は、根拠と再現手順を提示。
- 決定事項はShared Stateに記録し、次フェーズへ反映。

### 2.4 エスカレーションルール

- スコープ逸脱、重大リスク、テスト失敗が継続する場合はPlannerへ差戻し。
- 破壊的変更や運用影響が大きい場合はReleaseへ事前通知。

## 3. Maintenance and Operations

### 3.1 自動トリアージ

- 監視/ログ/アラートから重要度を分類 (P0/P1/P2)。
- 影響範囲と再現性の有無を判定し、優先度を決定。

### 3.2 修正提案フロー

- 問題の要約、影響範囲、再現手順を明記。
- 最小修正案とロールバック案を併記。

### 3.3 リリースゲート

- 変更の性質に応じて、ステージング検証を必須化。
- 重要な回帰リスクがある場合は人手承認を要求。

### 3.4 監査・記録

- 修正理由、テスト結果、リリース判断を記録。
- 重大障害はポストモーテムを作成。

## 4. Autonomous Operations

### 4.1 リスクベースの自動修正

- 低リスク (設定変更、軽微なバグ修正) のみ自動適用。
- 中高リスクは人手承認を要求。

### 4.2 自動回帰/負荷テスト

- 変更範囲に応じて最小テストセットを自動選定。
- 負荷影響が見込まれる変更は負荷テストを追加。

### 4.3 ガードレール

- SLO違反、コスト超過、セキュリティ重大影響は自動停止。
- ロールバック手順と実行権限を明確化。

### 4.4 運用メトリクス

- MTTR、失敗率、デプロイ頻度、コスト上限を監視。
- 閾値超過時は自動的にフェーズを引き下げる。
